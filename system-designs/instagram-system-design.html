📷 Designing Instagram

A step-by-step system design walkthrough for beginners — one concept at a time

Start Here

What we're building & how this guide works

We'll design a simplified Instagram. Instead of dumping the final architecture on you, we start with the simplest possible design — one that a beginner could build in a weekend — and then break it on purpose, step by step. Every time it breaks, we add exactly one new concept to fix it. By the end, the "scary" final diagram will feel obvious.

Meet our users (we'll follow them the whole way)

  • Priya — a regular user with 300 followers. She posts vacation photos.
  • Rahul — Priya's friend. He mostly scrolls his feed; he rarely posts.
  • Zayn — a celebrity with 50 million followers. He posts once a day.

The requirements (what the interviewer gave us)

Functional (what it does)Non-functional (how well it does it)

News Feed of top photos from people you followOK if a new photo shows up a bit late for some users (eventual consistency)
Highly reliable — an uploaded photo must never be lost
💡 Key insight to state early in the interview

Instagram is read-heavy. For every photo Priya uploads, thousands of people like Rahul view photos. A common assumption is a 100:1 read-to-write ratio. This single fact shapes almost every decision we'll make: we optimize for fast reads (viewing feeds) and can tolerate slightly slower writes (uploads).

Quick back-of-the-envelope math (interviewers love this)

Assume 500M total users, 1M daily active users uploading 2 photos/day, average photo 200 KB:

  • Uploads: 2M photos/day ≈ 23 photos per second
  • New storage: 2M × 200 KB = 400 GB per day ≈ 146 TB per year
  • Feed reads: with a 100:1 ratio, ~2,300 read requests/sec — and much more at peak

Conclusion: storage grows forever (we need cheap, infinitely-scalable photo storage), and reads dominate (we need caching). Keep these two in your pocket — they justify everything that follows.

Step 1

The simplest thing that works: one server

Start every design with the naive version. One machine runs our application code and our database.

📚 Concept: Client & Server

The client is the Instagram app on Priya's phone. The server is a computer in a data center running our code. They talk over the internet using APIs — named requests like uploadPhoto() or getFeed(). Think of an API like a restaurant menu: the client can only order things on the menu, and the kitchen (server) prepares them.

🎬 Real-world walkthrough

Priya taps "Share" on a beach photo. Her app sends the photo bytes to our server. The server saves the photo file to its hard disk and writes a row in its database: "photo #123, owner: Priya, caption: 'Goa!', time: 9:01am". When Rahul opens his app, it calls getFeed(); the server looks up who Rahul follows, finds Priya's photo, reads it from disk, and sends it back. Done!

Why this breaks (and it will)

⚠ Problems
  • Single point of failure: if this one server crashes, Instagram is down for everyone. Violates our "highly available" requirement.
  • It can't keep up: one machine can handle maybe a few thousand users. We have millions.
  • Disk fills up: 400 GB of new photos per day. A single machine's disk is full within days.
  • Photos can be lost: if the disk dies, Priya's photos are gone forever. Violates "never lose a photo."
✓ Takeaway

The single-server design is our baseline. Each next step fixes exactly one of these problems.

Step 2

Split photos from data: Object Storage

First fix: photos are huge compared to everything else. A photo is ~200 KB; the text about it ("Priya, 9:01am, caption 'Goa!'") is ~200 bytes — a thousand times smaller. These two kinds of data have totally different needs, so we store them in different places.

📚 Concept: Metadata vs. the actual file

Metadata means "data about data." For a photo, the metadata is: who posted it, when, the caption, and where the actual image file lives. Think of a library: the card catalog (metadata) tells you which shelf a book is on; the shelf (object storage) holds the actual book. You'd never stuff the whole book inside the card catalog.

📚 Concept: Object Storage (e.g., Amazon S3)

Object storage is a service built for storing files ("objects") — practically unlimited space, very cheap, and it automatically keeps 3+ copies of every file on different machines. If one disk dies, your photo survives. This is exactly how we satisfy "a photo must never be lost." You store a file and get back a URL like s3://photos/priya/123.jpg.

🎬 Real-world walkthrough

Priya uploads her beach photo. The server now does two things: (1) pushes the image file into object storage, which replies "saved at photos/priya/123.jpg, copied to 3 machines"; (2) writes one small row into the database: photo #123, owner Priya, caption "Goa!", url photos/priya/123.jpg. When Rahul views it, the server reads the tiny metadata row, and Rahul's phone downloads the image straight from object storage.

Why not store images inside the database?

  • Databases are optimized for small, structured rows — stuffing 200 KB blobs in makes every operation slow and backups enormous.
  • Object storage is ~10x cheaper per GB and scales to exabytes without us doing anything.
  • Serving a file from object storage doesn't waste our database's precious capacity.
✓ Takeaway

Separate big dumb files (object storage) from small smart data (database). Object storage's built-in replication already satisfies our "never lose a photo" requirement. ✨

Step 3

Millions of users: many servers + a Load Balancer

One app server can handle maybe a few thousand simultaneous users. We have millions. And if that single server crashes, everyone is locked out. The fix: run many identical copies of our app server.

📚 Concept: Horizontal vs. Vertical scaling

Vertical scaling = buy a bigger machine (more CPU/RAM). Simple, but there's a ceiling — you can't buy an infinitely big computer, and it's still one machine that can crash. Horizontal scaling = add more ordinary machines. No ceiling, and losing one machine is no big deal. Big systems almost always scale horizontally.

📚 Concept: Load Balancer

If we have 20 app servers, which one should Priya's phone talk to? A load balancer is a traffic cop that sits in front of them. Every request goes to the load balancer, which forwards it to a healthy, not-too-busy server (e.g., simple round-robin: server 1, 2, 3, 1, 2, 3...). It also runs health checks — pinging each server every few seconds — and stops sending traffic to any server that doesn't respond.

Restaurant analogy: the load balancer is the host at the door who seats each arriving party at whichever table (server) is free — and stops seating people at a broken table.

🎬 Real-world walkthrough

It's New Year's Eve and everyone is posting. Requests flood in; the load balancer spreads them across 50 app servers. At 12:03am, server #7 crashes. The load balancer's health check notices within seconds and simply stops routing to it. Rahul, mid-scroll, notices nothing — his next request just lands on server #8. That's high availability in action.

📚 Concept: Stateless servers

This only works if app servers are stateless — they remember nothing between requests (all data lives in the database/object storage). Then any server can handle any user's request, and adding/removing servers is trivial. Like fast-food cashiers: any cashier can take any customer's order because the menu and kitchen are shared.

⚠ New problem we just created

50 app servers can handle the traffic... but they all still talk to one database. The database is now the bottleneck and the single point of failure. That's our next two steps.

Step 4

What's in the database? Tables & the SQL/NoSQL choice

Before scaling the database, let's define what we actually store. We only need three core "tables" (think: spreadsheets):

🎬 Real-world walkthrough

When Rahul taps "Follow" on Priya's profile, we insert one tiny row: (rahul_id, priya_id) into the Follow table. When we later build Rahul's feed, we ask: "give me all followee_ids where follower_id = rahul" → [Priya, Zayn, ...], then "give me recent photos where owner is in that list." That's the entire logical heart of Instagram — two lookups!

📚 Concept: SQL vs. NoSQL

SQL databases (MySQL, PostgreSQL) store data in structured tables with strict rules and are great at combining tables ("joins"). NoSQL databases (Cassandra, DynamoDB) are more like giant distributed key–value stores: you look things up by key, they scale horizontally very easily, and they're built for high availability — but they're weaker at complex cross-table queries.

For us: our data is naturally relational (users, follows, photos), but our queries are simple key lookups, our scale is huge, and our requirements say availability over consistency. Both answers are defensible in an interview. A common, safe answer: SQL (e.g., MySQL) for user/follow data (small, relational), and a NoSQL wide-column store (e.g., Cassandra) for the huge, ever-growing photo metadata. What matters is that you justify the choice with the requirements.

📚 Concept: Index

An index is like the index at the back of a textbook: instead of scanning every page (row) to find "photos by Priya," the database keeps a sorted lookup structure on owner_user_id and jumps straight there. We'd index Photo.owner_user_id, Photo.created_at, and Follow.follower_id — the columns our feed queries filter on. Indexes make reads fast at the cost of slightly slower writes (the index must be updated too) — a great trade for our read-heavy app.

✓ Takeaway

Three tables — User, Photo, Follow — plus indexes on the columns we search by. The photo file itself is NOT here; only its URL is. Now let's make this database survive failures and scale.

Step 5

Scaling the database: Replication & Sharding

Two separate problems, two separate tools. Problem A: the database can die (availability). Problem B: it can't hold all the data or handle all the traffic (scale).

Tool A: Replication — copies for safety and read speed

📚 Concept: Primary–Replica replication

Keep several copies of the database. All writes (new photos, new follows) go to one primary; it continuously streams every change to several replicas (copies). Reads can be served by any replica. Benefits: (1) if the primary dies, a replica is promoted and we keep running — availability ✓; (2) since Instagram is 100:1 read-heavy, spreading reads across replicas multiplies our read capacity ✓.

Writes → primary. Reads → replicas. If primary dies, a replica takes over.
🎬 Real-world walkthrough — where "eventual consistency" comes from

Priya posts her photo at 9:01:00. It's written to the primary and reaches Replica 1 at 9:01:01, Replica 2 at 9:01:02. Rahul refreshes at 9:01:01 and his read happens to hit Replica 2 — he doesn't see the photo yet! He refreshes again at 9:01:05 and there it is. That tiny delay is eventual consistency, and our requirements explicitly said this is fine. We chose availability over perfect freshness — this is the classic CAP theorem trade-off (in a distributed system, when failures happen, you must pick availability or strict consistency; we pick availability).

Tool B: Sharding — splitting the data itself

📚 Concept: Sharding (partitioning)

Replication copies the same data everywhere — it doesn't help when the data is too big for one machine. Sharding splits the data into pieces, each on a different machine. Like a library that outgrew one building and split into branches: A–H in building 1, I–P in building 2, Q–Z in building 3. A rule (the shard key) tells you which branch holds what.

For photo metadata, shard by photo_id: e.g., shard = photo_id % 10 spreads photos evenly across 10 database machines. (Sharding by user is tempting but risky: Zayn the celebrity would make one shard extremely "hot" while others idle.)

Bonus: how do we generate photo IDs if there's no single database?

With many shards, we can't rely on one database's auto-incrementing counter. Common answers: a dedicated ID-generation service, or IDs that encode a timestamp + machine ID + counter (like Twitter's "Snowflake"). Timestamp-first IDs are handy: sorting by ID = sorting by time, perfect for newest-first feeds.

✓ Takeaway

Replication = same data, many copies → survives failures, multiplies read capacity. Sharding = different data, many machines → unlimited size. Real systems use both together.

Step 6

Making reads fast: Cache & CDN

Our 200ms feed budget is tight. A database query might take 10–50ms, and building a feed needs several. Photos coming from far-away storage can take even longer. Two tools fix this — both are forms of the same idea: keep popular stuff close by.

Tool A: Cache (for metadata)

📚 Concept: Cache (e.g., Redis / Memcached)

A cache is a small, blazing-fast storage layer that keeps data in RAM (memory) instead of on disk. Reading from RAM takes ~0.1ms vs ~10ms+ for a database query — up to 100x faster. It's your kitchen counter vs. the supermarket: you keep the ingredients you use daily on the counter instead of driving to the store every meal.

The standard pattern is cache-aside: when the app needs data, it (1) checks the cache — if found, that's a cache hit, done in a millisecond; (2) if not (cache miss), it queries the database, returns the result, and puts a copy in the cache for next time.

📚 Concept: The 80/20 rule & LRU eviction

Roughly 20% of content generates 80% of the traffic (everyone's looking at the same trending posts). So a cache holding just the hot 20% absorbs most reads. The cache is small, so when it fills up we evict the Least Recently Used (LRU) items — exactly like clearing the pantry of things you haven't touched in months.

🎬 Real-world walkthrough

Zayn posts a photo. The first fan to view it triggers a cache miss → database query (~20ms) → result stored in Redis. The next 10 million fans get cache hits at ~0.5ms each. Our database barely notices that 10 million people viewed the post. Without the cache, it would have melted.

Tool B: CDN (for the photos themselves)

📚 Concept: CDN — Content Delivery Network

A CDN (Cloudflare, CloudFront, Akamai) is a network of thousands of servers spread around the world that cache files — our photos. Speed of light is real: if our object storage is in Virginia and Rahul is in Mumbai, every image fetch crosses the planet (~200ms+ just in travel time — our whole budget!). With a CDN, the first Mumbai viewer pulls the photo from Virginia, and the CDN's Mumbai edge server keeps a copy. Every later viewer nearby gets it in ~20ms.

Analogy: instead of one warehouse in the US shipping worldwide, you place local convenience stores in every city stocked with the popular items.

✓ Takeaway

Cache (Redis) makes metadata reads ~100x faster; CDN makes photo downloads fast worldwide. Together they're how a 100:1 read-heavy system stays under 200ms. One piece left — the trickiest one: actually generating the feed.

Step 7 — the heart of the problem

Generating the News Feed: Pull vs. Push vs. Hybrid

This is usually the deep-dive of the interview. The question: when Rahul opens the app, how do we assemble "recent top photos from everyone he follows" in under 200ms?

Approach 1: Pull ("build it when he asks")

When Rahul opens the app: look up who he follows (say 500 people), query recent photos for each, merge and sort by time, return the top ~50.

⚠ Why pull alone is too slow

That's querying photos of 500 users across multiple shards, then sorting — potentially hundreds of milliseconds of work while Rahul stares at a spinner. It also repeats this work on every single refresh, mostly recomputing the same answer. Blows the 200ms budget.

Approach 2: Push / "fan-out on write" ("build it before he asks")

📚 Concept: Precomputed feeds (fan-out on write)

Flip it around: maintain a ready-made feed list for every user, stored in the cache/database (just a list of photo IDs, e.g. "Rahul's feed: [photo 981, 977, 970…]"). When Priya posts, a background job finds her 300 followers and inserts the new photo ID into each of their 300 feed lists. Now when Rahul opens the app, we just read his precomputed list — one cache read, ~1ms. The 200ms budget is easy.

Analogy: pull = going to 500 shops each morning to see what's new. Push = every shop drops a flyer in your mailbox the moment they get new stock; you just open your mailbox.

🎬 The celebrity problem

Push works beautifully for Priya (300 followers = 300 quick inserts). But Zayn has 50 million followers. One tap on "Share" would trigger 50 million writes! That could take many minutes and hammer our systems — and most of those followers aren't even online.

Approach 3: Hybrid (the interview answer)

  • Normal users (99.9%): push. Their posts fan out to followers' precomputed feeds instantly.
  • Celebrities (followers above some threshold, e.g. 100k): don't fan out. Instead, when Rahul opens his app, we pull just from the handful of celebrities he follows and merge their fresh posts with his precomputed feed.

Rahul follows 499 normal users (precomputed ✓) and 1 celebrity (one quick extra query ✓). Best of both worlds.

📚 Concept: Message Queue (async processing)

Fan-out shouldn't happen while Priya waits for her upload to finish. Instead the app server drops a small message — "Priya posted photo #123" — onto a message queue (Kafka, RabbitMQ, SQS), tells Priya "posted!" immediately, and separate fan-out worker machines consume the queue and update follower feeds in the background over the next second or two. A queue is a to-do list between systems: the waiter clips your order to the kitchen rail and moves on; cooks work through the rail at their own pace. This is also why a follower might see the photo a few seconds "late" — and our requirements said that's fine (eventual consistency again).

Pull = compute on read (slow reads, cheap writes). Push = compute on write (instant reads, expensive writes). Hybrid: push for normal users, pull for celebrities. Do fan-out asynchronously via a message queue. This is THE key discussion in the interview — know it cold.

Step 8

Never lose a photo, never go down

Let's explicitly verify the two strictest requirements.

"Any uploaded photo should never be lost" (durability)

  • Object storage keeps 3+ copies of every photo on different machines — often in different buildings (availability zones). One disk, one server, even one building failing loses nothing.
  • The upload API only tells Priya "posted ✓" after object storage confirms the file is safely replicated. If anything fails mid-upload, she sees an error and retries — we never silently drop a photo.
  • Metadata is equally protected: every database shard has replicas, plus periodic backups.

"Highly available" (the system stays up)

📚 Concept: Redundancy — no single point of failure

Walk the diagram and ask of every box: "what if this machine dies right now?" Every answer must be "a copy takes over."

  • App server dies → load balancer routes around it (Step 3).
  • Load balancer dies → yes, even it has a standby twin that takes over its address.
  • Database primary dies → a replica is promoted to primary (Step 5).
  • Cache dies → not fatal: requests fall through to the database (slower, but up). Caches are also clustered.
  • Whole data center dies → big systems run in multiple geographic regions; traffic shifts to another region.
🎬 Real-world walkthrough

At 2am a power failure takes out the building holding our primary metadata database for shard 3. Within seconds: monitoring detects it, replica in another building is promoted to primary, app servers reconnect. Rahul, doom-scrolling at 2am, might see one refresh take an extra second. Priya's photos? All safe — they exist in 3 places. This is what "designing for failure" means: assume every part WILL fail, and make failure boring.

Separating reads from writes (a nice extra point)

Uploads (writes) and feed-viewing (reads) have different shapes — writes are few but heavy (big files), reads are many but light. Many designs use separate upload servers and read servers so a burst of uploads never slows down everyone's scrolling. This is the same "specialize each part" instinct as splitting photos from metadata.

✓ Takeaway

Durability = replicate the data (3+ copies, confirm before acknowledging). Availability = replicate the machines (redundancy at every layer). The price is eventual consistency — which our requirements happily accept.

Flow 1 — Priya uploads a photo (write path)

  1. Her app → load balancer → an upload server.
  2. Server stores the image in object storage (replicated 3+ times) — gets back a URL.
  3. Server writes metadata (owner, caption, URL, time) to the correct database shard.
  4. Server drops "Priya posted #123" on the message queue and immediately tells Priya "✓ Posted".
  5. Fan-out workers read the queue, look up Priya's 300 followers, insert #123 into each of their feed cache lists. (If Zayn posted, we'd skip this — he's over the celebrity threshold.)

Flow 2 — Rahul opens his feed (read path, <200ms)

  1. His app → load balancer → a read server.
  2. Server grabs Rahul's precomputed feed list from the feed cache (~1ms): [#123, #981, …].
  3. It pulls recent posts from the 1 celebrity Rahul follows (Zayn) and merges them in.
  4. It fetches photo metadata for those IDs — mostly from the metadata cache, database on misses.
  5. Rahul's phone downloads the actual images from the nearest CDN edge.

How to narrate this in a 45-minute interview

  1. (5 min) Clarify requirements; state read-heavy 100:1; do the storage/QPS math.
  2. (5 min) Draw the simple high-level design (client → LB → app servers → DB + object storage).
  3. (10 min) Data model: three tables, SQL vs NoSQL reasoning, indexes.
  4. (15 min) Deep dives: news feed pull/push/hybrid (the star topic), then sharding, caching, CDN.
  5. (5 min) Reliability: replication, no single point of failure, walk the "what if X dies?" game.
  6. (5 min) Trade-offs & extensions: consistency vs availability choice, celebrity problem, ranking feeds by ML, video support.
Quick glossary — every concept in one line
  • Load balancer: traffic cop distributing requests across identical servers.
  • Horizontal scaling: add more machines instead of bigger machines.
  • Stateless server: keeps no memory between requests, so any server can serve anyone.
  • Object storage: cheap, infinite, auto-replicated file storage (S3).
  • Metadata: data about data — caption/owner/URL, not the image itself.
  • Replication: same data copied to several machines (safety + read capacity).
  • Sharding: data split across machines by a key (size + write capacity).
  • Index: textbook-index lookup structure making queries fast.
  • Cache: hot data in RAM; ~100x faster than disk; LRU eviction; 80/20 rule.
  • CDN: worldwide cache of files, served from near the user.
  • Message queue: to-do list between systems enabling background (async) work.
  • Fan-out on write: push new post into every follower's precomputed feed.
  • Eventual consistency: everyone sees the data eventually, maybe not instantly.
  • CAP theorem: under failure, pick availability or strict consistency — we chose availability.
