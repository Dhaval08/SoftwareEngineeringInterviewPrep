💬 Designing Slack

A step-by-step system design walkthrough — focus on real-time messaging, the new muscle Instagram & Twitter didn't exercise

Start Here

What's fundamentally different about Slack

Instagram and Twitter are pull products: you open the app and ask for content. Slack is a push product: a message must appear on your screen the instant a coworker sends it, without you asking. That single difference — the server needs to talk to the client whenever it wants — reshapes the whole design. This guide's deep dive is exactly that: how does a server push data to millions of connected clients in real time?

Meet our cast (a workspace called "Acme Corp")

  • Maya — engineer, sends a message in the #engineering channel.
  • Rohan & Dev — her teammates, online, both looking at #engineering right now.
  • Priya — a teammate whose laptop is closed; she's offline and should get a notification + see the message when she returns.
  • #general — a channel with 8,000 members (the "big channel" problem).

💡 Key insights to state early
  • The hard part isn't storing messages — it's delivering them. Saving a message to a database is easy (we've done it twice). Pushing it to everyone watching that channel, right now, is the challenge.
  • Ordering matters more than in social apps. A conversation reads wrong if messages arrive out of order. We need a consistent per-channel ordering.
  • Connections are the scarce resource. With social apps we counted requests/sec; here we count how many clients are holding an open connection at once — potentially tens of millions.

Back-of-the-envelope

  • Say 10M simultaneously-connected users. Each holds one persistent connection → 10M open connections to keep alive.
  • Messages are tiny (~a few hundred bytes). Storage is cheap, like Twitter. The cost is the delivery fan-out, not bytes.
  • A message in a 5-person channel = 4 deliveries. A message in #general (8,000 members) = up to 8,000 deliveries. Big channels are our celebrity problem.
Step 1

First attempt: why "just refresh" (polling) fails

The skeleton (load balancer → stateless app servers → sharded/replicated DB → object storage/CDN for file uploads) carries over from the earlier guides — we'll take it as given. The interesting question is delivery. Naive idea: the client asks "any new messages?" over and over.

📚 Concept: Polling

Polling = the client repeatedly sends a normal request ("anything new since message #500?") every, say, 3 seconds. It's simple and it's how you'd first try. Analogy: a kid in the back seat asking "are we there yet?" every few seconds the whole trip.

⚠ Why polling breaks for chat
  • Laggy: if Maya sends a message right after Rohan's poll, he waits up to 3 seconds to see it. Chat needs to feel instant.
  • Wasteful: 10M clients each polling every 3s = ~3.3M requests/sec, and the vast majority return "nothing new." You're paying enormous cost to mostly say "no."
  • Poll faster to reduce lag → cost explodes; poll slower to save cost → more lag. There's no good setting. The model is wrong for real-time.

The root problem: in a normal request the client must initiate, but with chat it's the server that knows when there's news. We need to invert who starts the conversation.

✓ Takeaway

Request/response (the model behind Instagram & Twitter) can't push. We need a connection the server can send down at any moment. Enter WebSockets.

Step 2 — the core idea

Persistent connections: WebSockets

📚 Concept: WebSocket (a persistent, two-way pipe)

A normal web request is like sending a letter: you write, you send, you get one reply, the exchange is over. A WebSocket is like keeping a phone line open: the client dials in once, and the line stays connected. Now either side can speak whenever they want — no need to re-dial. The server can push "new message!" down the line the instant it happens.

When Rohan opens Slack, his app makes one WebSocket connection to a server and just holds it open for hours. Maya's message travels: Maya → server → straight down Rohan's already-open pipe → on his screen in ~50ms. No polling, no waiting.

📚 Concept: Gateway / connection servers

We dedicate a fleet of servers whose only job is to hold these open connections — call them gateway servers (or "WebSocket servers"). Each can hold maybe ~500k–1M open connections. With 10M users online, that's ~10–20 gateway machines' worth of connections. They're specialized: they don't run business logic, they just keep pipes alive and shuttle messages in and out. (Same instinct as separating photos from metadata — give each job its own optimized box.)

⚠ The problem this creates — and it's the crux

Rohan is connected to Gateway 1. Dev is connected to Gateway 2. Maya sends to #engineering. The server now must figure out: which gateways hold connections for members of this channel, so I can push the message to exactly them? Rohan's message must reach Gateway 1, Dev's must reach Gateway 2 — but the message might first arrive at some third server. How do the servers coordinate? That's Step 3.

Step 3 — the heart of the design

Routing a message to the right connections: Pub/Sub

Maya sends a message. Members of #engineering are scattered across many gateway servers. We need a way to deliver to "everyone subscribed to this channel" without every server knowing about every other server. The tool is Publish/Subscribe.

📚 Concept: Pub/Sub (publish–subscribe)

Think of a radio station. The DJ (publisher) broadcasts on 98.5 FM. Anyone tuned to 98.5 (subscribers) hears it. The DJ doesn't know or care who's listening — it just broadcasts to the channel. Pub/Sub systems (Redis Pub/Sub, Kafka) work the same way: publishers send messages to a named topic, and every server subscribed to that topic receives them.

We make each Slack channel a Pub/Sub topic. A gateway subscribes to a channel's topic when it's holding a connection for someone viewing that channel. To deliver a message, we just publish it to the channel's topic — and every gateway with relevant users automatically gets it and pushes it down the right pipes.

🎬 Real-world walkthrough — one message's journey

Maya types "deploy is green ✅" in #engineering and hits enter.

  1. Her message travels up her open WebSocket to the gateway she's on, then to the Message Service.
  2. The service saves it to the database first (durability — see Step 4), assigns it an ID/sequence number, and acknowledges Maya (her client shows the "sent" checkmark).
  3. It publishes the message to topic chan:engineering.
  4. Gateway 1 (subscribed, because Rohan is viewing the channel) receives it and pushes it down Rohan's pipe. Gateway 2 does the same for Dev. Both see it within ~50ms.
  5. Priya is offline — no open connection — so nothing is pushed to her now. Instead a notification service (also subscribed) sends her a mobile push, and her unread count for #engineering is bumped. When she reopens Slack, her client fetches messages it missed from the database (Step 4).
✓ Takeaway

Pub/Sub decouples senders from receivers: publish to a channel topic, and gateways fan the message out to exactly the right open connections. Online users get a real-time push; offline users get a notification + catch up later. Now let's make sure messages are never lost and always in order.

Step 4

Never lose a message; keep everyone's view in order

Store first, then deliver

Notice the ordering in the walkthrough: we save to the database before publishing. This is deliberate. If we pushed to everyone first and the save failed, online users would see a message that vanishes on refresh, and offline users would never get it. Saving first means the database is the single source of truth; real-time delivery is a fast bonus layer on top.

📚 Concept: Data model & sharding by channel

We shard messages by channel_id, so all of a channel's messages live together on one shard — fast to load history, and a natural place to assign a strict per-channel sequence number. Loading a channel = "give me messages for channel X with seq > my last_read," a clean indexed query.

📚 Concept: Message ordering

If Maya and Dev send at nearly the same instant, everyone must see them in the same order, or the conversation reads differently for different people. Solution: the shard that owns a channel assigns each message a monotonically increasing sequence number as it's saved. Clients render strictly by sequence. Because one shard owns the channel, there's a single authority deciding "this was message #841, that was #842" — no ambiguity. (This is why sharding by channel, not by message, matters here.)

📚 Concept: Client catch-up & delivery guarantees

Networks drop. A pushed message can be missed if a connection blips at the wrong moment. So the client never fully trusts the live stream: it tracks the last sequence number it has, and on (re)connect it asks "give me everything in this channel after seq 840." Missed messages are backfilled from the database. This gives at-least-once delivery; the sequence number lets the client discard duplicates → effectively exactly-once as the user sees it. Real-time push is the fast path; the database + sequence number is the safety net that guarantees no loss.

🎬 Real-world walkthrough

Rohan's train enters a tunnel; his WebSocket drops for 90 seconds. Maya posts 3 messages meanwhile. When Rohan reconnects, his client reopens a WebSocket, tells the server "my last #engineering seq is 840," and the server returns messages 841–843 from the database. Rohan never notices a gap — the live pipe and the durable store together cover for each other.

✓ Takeaway

Persist before you push. Shard by channel so one authority assigns sequence numbers → consistent ordering. Clients catch up by sequence on reconnect → no message ever lost, even across flaky networks.

Step 5

Presence (who's online) & the big-channel problem

Presence: the green dot

📚 Concept: Presence via connection state + heartbeats

Who's online? Naturally, whoever currently holds an open WebSocket. Gateways know this, so they write "user X is online" into a fast in-memory store (Redis). Clients send a periodic heartbeat ("still here") every ~30s; if the gateway stops hearing it, it marks the user offline. Presence is intentionally eventually consistent — if Dev's green dot lingers a few seconds after he closes his laptop, nobody's harmed. We deliberately spend less rigor here than on messages.

⚠ Presence can be a hidden scale trap

Broadcasting "Dev is online!" to everyone who could see Dev can be more traffic than actual messages. Fix: only compute/emit presence for people currently looking at a relevant channel or DM, and batch updates. Interviewers like to poke at this — presence looks trivial but fans out badly.

The big-channel problem (our "celebrity" of Slack)

🎬 The #general problem

A message in a 5-person channel is 4 deliveries — trivial. A message in #general with 8,000 members, during an all-hands, is up to 8,000 near-simultaneous pushes. If a CEO posts and everyone reacts, you get thundering fan-out. This is the same shape as Twitter's celebrity problem, now in delivery form.

  • Pub/Sub already helps: we publish once to the channel topic; only gateways holding a member's connection do work, and each pushes only to its own local connections. No server fans out to all 8,000 alone.
  • Only deliver to who's watching: most of the 8,000 aren't looking at #general right now. We push in real time to active viewers; everyone else just gets an unread-count bump and reads it later from the DB. Delivery scales with active viewers, not total members.
  • Rate-limit & coalesce: for reactions/typing indicators in huge channels, batch updates ("+142 👍") rather than 142 individual events.
✓ Takeaway

Presence = connection state + heartbeats, kept cheap and eventually consistent. Big channels are tamed by publishing once (Pub/Sub), delivering live only to active viewers, and batching noisy events.

Narrating this in a 45-minute interview

  1. (5 min) Requirements + "the hard part is delivery, not storage"; count connections, not just QPS.
  2. (5 min) Kill polling; motivate persistent connections.
  3. (8 min) WebSockets + dedicated gateway fleet; surface the cross-gateway routing problem.
  4. (12 min) Pub/Sub deep dive: channel topics, publish-once fan-out, online vs offline paths.
  5. (8 min) Durability & ordering: persist-before-publish, per-channel sequence numbers, client catch-up, at-least-once → exactly-once.
  6. (4 min) Presence + big-channel fan-out.
  7. (3 min) Trade-offs: strict ordering/no-loss for messages vs eventual consistency for presence; gateway failure handling.
New glossary entries (beyond the earlier guides)
  • Polling: client repeatedly asks "anything new?" — simple but laggy and wasteful for chat.
  • WebSocket: a persistent, two-way connection; either side can send anytime (the open phone line).
  • Gateway / connection server: specialized server that just holds millions of open connections.
  • Pub/Sub: publish to a named topic; all subscribers receive it (the radio station).
  • Topic: a named channel in Pub/Sub — here, one per Slack channel.
  • Sequence number: per-channel counter giving every member the same message order.
  • Heartbeat: periodic "still here" ping used for presence and detecting dead connections.
  • At-least-once / exactly-once: delivery guarantees; sequence numbers let clients dedupe to exactly-once as seen.
  • Persist-before-publish: save to the durable store before doing real-time delivery, so nothing is lost.
