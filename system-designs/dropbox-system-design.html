📁 Designing Dropbox

A step-by-step system design walkthrough — focus on file sync: chunking, dedup, delta sync & conflict resolution

Start Here

What's fundamentally different about Dropbox

The earlier apps served content to phones. Dropbox's job is stranger: keep a folder identical across many of your devices and collaborators, automatically. You edit a file on your laptop; seconds later it's updated on your phone, your desktop, and your teammate's machine. The magic word is sync, and getting it right — fast, cheap on bandwidth, and never corrupting or losing a file — is the whole game. That's this guide's deep dive.

💡 Key insights to state early
  • Split the problem in two: file contents (big, dumb bytes) and file metadata (small facts: name, folder, version, which chunks). These want totally different systems — the same "big files vs. small smart data" split from the Instagram guide, taken further.
  • Never re-upload a whole file for a tiny edit. Change one slide in a 50 MB deck and we should transfer only the changed part. This is delta sync, and it's the signature Dropbox trick.
  • Durability beats everything. Users trust Dropbox with their only copy of important files. Losing or corrupting data is the one unforgivable failure.

Back-of-the-envelope

  • Say 500M users, average 50 GB stored each → hundreds of petabytes. Storage is the dominant cost → dedup & cheap object storage matter enormously.
  • Most "syncs" are small edits, not new files → optimizing the edit path (delta sync) saves the most bandwidth.
  • Read/write ratio is more balanced than social apps, but heavily favors "nothing changed" — clients mostly ask "any updates?" and hear "no."
Step 1

The naive version: whole-file upload & download

Simplest thing that works: the client watches your folder; when a file changes, it uploads the entire file to the server, which stores it. Other devices poll "is there a newer version?" and download the whole thing.

⏪ Recap: the skeleton (from earlier guides)

Clients → load balancer → stateless app servers → a database (replicated + sharded) for metadata and object storage (S3, 3+ replicated copies) for the file bytes. We already know object storage gives us cheap, effectively-infinite, durable file storage — that's the backbone of "never lose a file."

⚠ Why whole-file sync is painful
  • Wasteful: Maya fixes one typo in her 50 MB deck → we re-upload all 50 MB. Multiply across millions of tiny edits = staggering wasted bandwidth and money.
  • Fragile on big files: uploading a 2 GB file over hotel wifi that drops at 90% means starting over from zero.
  • Slow to sync: her desktop must download the whole 50 MB to get that one-character fix.
  • Duplicate storage: if 10,000 employees have the same 20 MB company handbook, we store it 10,000 times.
✓ Takeaway

Whole-file sync works but bleeds bandwidth and storage. The fix to nearly all of these problems is one idea: stop treating a file as one blob. Break it into chunks.

Step 2 — the core idea

Break files into chunks

📚 Concept: Chunking

Instead of storing a file as one big blob, the client slices it into fixed-size pieces — chunks (Dropbox uses ~4 MB). A 50 MB file becomes ~13 chunks. Each chunk is stored separately in object storage, and the file's metadata records the ordered list of chunks that reassemble it. Think of a book split into chapters: to fix a typo in chapter 7, you reprint only chapter 7, not the whole book.

📚 Concept: Content hashing (the chunk's fingerprint)

Each chunk gets a hash — a short fingerprint computed from its bytes (e.g. SHA-256). Identical bytes always produce the same fingerprint; change one byte and the fingerprint changes completely. We name and look up chunks by their hash. This fingerprint is the key that unlocks both delta sync and dedup (next steps).

A file = an ordered list of hashed chunks. Metadata stores the list; object storage stores the chunks.
📚 Concept: Metadata vs. block storage (the two-database split)

This makes the split from the Instagram guide crisp. We keep two systems:

  • Metadata Service + DB — small, structured facts: files, folders, versions, and each file's ordered chunk-hash list, permissions, sharing. This is the "brain."
  • Block Storage — the actual chunk bytes, keyed by hash, sitting in object storage (3+ copies). This is the "warehouse."

The metadata is tiny but queried constantly; the blocks are huge but simple. Specializing each is why the system scales.

✓ Takeaway

Chunk the file, fingerprint each chunk by content hash, store chunks in block storage and the chunk-list in a metadata DB. Everything good below flows from this one decision.

Step 3 — the payoff

Delta sync & deduplication

Delta sync — only move what changed

📚 Concept: Delta sync

When Maya edits her deck, the client re-chunks the file and compares the new chunk fingerprints against the old ones. Almost all chunks are identical (same hash) — only the chunk(s) she touched have new hashes. The client uploads only the changed chunks and tells the metadata service "the new version of report.pptx is: [same, same, NEW, same, …]". One edited slide → one 4 MB chunk uploaded instead of 50 MB. ~92% bandwidth saved.

before edit
c1c2c3c4c5…
Maya edits slide 3 → re-chunk → compare fingerprints
c1c2c3′ NEWc4c5…
Only c3′ is uploaded; every other chunk already exists in block storage

Deduplication — never store the same bytes twice

📚 Concept: Deduplication

Because chunks are addressed by content hash, two identical chunks have the same name — so they're automatically the same object in block storage. Before uploading a chunk, the client asks the metadata service "do you already have hash a3f9…?" If yes, it skips the upload entirely and just references it. Consequences:

  • The 20 MB company handbook shared by 10,000 employees is stored once, referenced 10,000 times.
  • Re-uploading a file you already have transfers nothing — this is why dragging a big file that's already in Dropbox finishes instantly.
🎬 Real-world walkthrough

Maya tweaks one chart in report.pptx and saves. Her client: (1) re-chunks — 13 chunks, 12 fingerprints unchanged, 1 new; (2) asks the metadata service which of the 13 hashes it already has → "I have 12, send me the 13th"; (3) uploads that one 4 MB chunk to block storage; (4) commits a new version pointing at the new chunk list. Total moved: 4 MB, not 50 MB. Her desktop, when it syncs, downloads only that same 4 MB chunk and reassembles the file locally. Fast for her, cheap for us.

✓ Takeaway

Content-hashed chunks give us delta sync (upload only changed chunks) and dedup (store identical chunks once) for free. Massive savings in bandwidth and storage — the two things Dropbox spends the most on. Bonus: chunking also makes resumable uploads trivial (a dropped connection only re-sends the unfinished chunk).

Step 4

How devices find out something changed

Delta sync explains how to move changes efficiently. But how does Maya's desktop learn there's a change to fetch in the first place? Two paths, and Dropbox famously uses both.

📚 Concept: Long-polling notification vs. metadata polling

Every online client keeps a lightweight connection to a Notification Service (long-polling or a WebSocket, like Slack's gateways). This channel carries no file data — just a tiny nudge: "something in your account changed, go check." On that nudge, the client calls the Metadata Service to ask "what's new since version 41?" and gets back the list of changed files/chunks to download. Splitting the cheap "wake up!" signal from the heavier "here's what changed" query keeps the always-on connections extremely light.

📚 Concept: Client-side watcher & local index

On each device the Dropbox client watches your folder for changes and keeps a local index (a little database of "which file is at which version, made of which chunk hashes"). When a file changes locally, the watcher notices, re-chunks, and pushes deltas up. When a remote change arrives, it downloads deltas and updates local files. The local index is what lets the client compute deltas and detect conflicts without re-reading everything.

✓ Takeaway

A light notification channel says "wake up," the metadata service says "here's exactly what changed," and only changed chunks travel. Clients keep a local index to drive it all. But what happens when two people change the same file at once? That's the last hard problem.

Step 5

Conflicts, versioning & never losing data

The conflict problem

🎬 Two edits collide

Maya edits report.pptx on her laptop offline on a flight. Meanwhile Rohan edits the same file and syncs it to the cloud. Maya lands, reconnects — now there are two different "next versions" of the same file, both based on version 41. Who wins? If we blindly overwrite, someone's work is destroyed — a cardinal sin for a product people trust with their files.

📚 Concept: Versioning & conflict resolution

The metadata service treats file history as a chain of versions. Each update says "I'm basing this on version 41." The first update to arrive becomes version 42. When the second update arrives also claiming to build on 41, the service detects the clash. Rather than silently pick a winner and lose data, Dropbox's pragmatic rule: keep both. It creates a conflicted copy — e.g. report (Maya's conflicted copy).pptx — so no edit is ever lost, and a human decides how to merge. Simple, safe, and honest about the fact that two people genuinely diverged.

Why not auto-merge like Google Docs? Because arbitrary binary files (a .pptx, a .zip) can't be merged line-by-line the way text can. "Keep both, let a human reconcile" is the correct, durable default for a general file store.

📚 Concept: File history & version restore

Because every version is just a different chunk list and old chunks are still in block storage (thanks to dedup, they're cheap to keep), we get version history and "restore previous version" almost for free — just point the file back at an earlier chunk list. This also protects against accidental deletes and ransomware: the old bytes are still there.

Durability — the sacred requirement

📚 Concept: Durability by replication + verification
  • Every chunk is stored with 3+ replicas across machines/buildings in object storage. A disk, server, or datacenter failure loses nothing.
  • The client only reports "synced ✓" after the metadata commit succeeds, so an interrupted upload never leaves a file half-recorded.
  • The content hash doubles as an integrity check: on download, the client re-hashes each chunk and confirms it matches — catching any silent corruption in transit or storage.
  • The metadata DB is itself replicated and backed up — losing the "brain" that maps files to chunks would be as bad as losing the chunks.
✓ Takeaway

Conflicts → keep both (conflicted copy), never overwrite. Versions → free from chunk lists + retained chunks. Durability → replicate chunks and metadata, ack only after commit, verify with hashes. Losing data is the one thing this system refuses to do.

Two worlds: the metadata "brain" (small, smart) and block storage "warehouse" (huge, simple)

Flow 1 — Maya edits report.pptx (upload path)

  1. Client watcher detects the save, re-chunks the file, hashes each chunk, compares against its local index → 1 new chunk.
  2. Client asks Metadata Service which hashes are new (dedup check) → uploads just that chunk via the Block Service to block storage (3+ copies).
  3. Client commits a new file version (new chunk list) to the Metadata Service; only now does it show "synced ✓".
  4. Metadata Service tells the Notification Service, which nudges Maya's other devices and Rohan (shared folder).

Flow 2 — Rohan's device receives the change (download path)

  1. Notification Service nudges Rohan's client: "updates available."
  2. Client asks Metadata Service "what changed since my version?" → gets the new chunk list.
  3. Client downloads only the chunks it doesn't already have (via CDN/block storage), verifies each by re-hashing, and reassembles the file locally.
  4. If Rohan had also edited it → conflict detected → a conflicted copy is created so no edit is lost.

New glossary entries (beyond the earlier guides)
  • Chunk: a fixed-size piece of a file (~4 MB); files are stored and synced as lists of chunks.
  • Content hash: a fingerprint of a chunk's bytes; same bytes → same hash. Used as the chunk's name.
  • Delta sync: transfer only the chunks that changed, not the whole file.
  • Deduplication: store identical chunks once, referenced many times.
  • Block/metadata split: chunk bytes in block storage; file facts & chunk lists in a metadata DB.
  • Resumable upload: because a file is chunks, a dropped connection only re-sends the missing chunk.
  • Local index: the client's little DB of file → version → chunk hashes; drives delta & conflict detection.
  • Conflicted copy: when two edits diverge, keep both files rather than overwrite → no data loss.
  • Integrity check: re-hash a downloaded chunk to confirm it isn't corrupted.
